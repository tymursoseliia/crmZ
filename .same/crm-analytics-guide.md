# 📊 Аналитика CRM - Руководство

## Обзор

Система аналитики CRM теперь включает полное отслеживание движения лидов по воронке продаж с детальной статистикой срезов (потерянных лидов) на каждом этапе.

## 🎯 Основные возможности

### 1. **История изменений статусов**
- Каждое изменение статуса лида записывается в историю
- Сохраняется: дата/время, предыдущий статус, новый статус, кто изменил
- История используется для аналитики срезов и времени на этапах

### 2. **Воронка продаж**
Визуальное представление количества лидов на каждом этапе:
- Договор сделан
- Дал реквизиты
- Оплата за растаможку
- Оплата за авто
- Оплата за утиль
- Оплата за госпошлину
- Оплата за залоговый платеж
- Оплата прочее
- Завершено

**Метрики:**
- Количество лидов на каждом этапе
- Сумма потенциальной выручки
- Конверсия между этапами (% перехода)

### 3. **Анализ срезов**
Показывает **откуда именно** лиды переходят в статус "Потеряно":
- Список этапов, на которых происходят потери
- Количество потерянных лидов с каждого этапа
- Процент от общих потерь

**Пример:**
```
Оплата за растаможку → Потеряно: 5 лидов (50% от всех потерь)
Оплата за авто → Потеряно: 3 лида (30% от всех потерь)
```

### 4. **Среднее время на этапах**
Рассчитывает среднее количество дней, которое лид проводит на каждом этапе:
- Помогает выявить узкие места в воронке
- Позволяет оптимизировать процессы
- Планировать сроки закрытия сделок

### 5. **Общая статистика**
- **Всего лидов** - общее количество в системе
- **Завершено** - успешно закрытые сделки + % конверсии
- **Потеряно** - срезанные лиды + % срезов
- **Выручка** - реальная + потенциальная

### 6. **Статистика по командам**
Отдельная аналитика для:
- Команды Вохи
- Команды Зета

## 🔄 Как работает отслеживание

### Создание лида
При создании нового лида автоматически создается первая запись в истории:
```typescript
{
  id: "hist-1234567890",
  timestamp: "2026-02-04T10:00:00Z",
  previousStage: null,
  newStage: "contract_done",
  changedBy: "manager-123",
  notes: "Создан лид"
}
```

### Изменение статуса
При каждом изменении статуса добавляется новая запись:
```typescript
{
  id: "hist-1234567891",
  timestamp: "2026-02-05T14:30:00Z",
  previousStage: "contract_done",
  newStage: "gave_requisites",
  changedBy: "teamlead-456",
  notes: "Выдал реквизиты клиенту"
}
```

### Анализ срезов
Когда лид переходит в статус "lost", система:
1. Находит последнюю запись в истории перед "lost"
2. Определяет с какого этапа произошел срез
3. Группирует статистику по этапам

## 📈 Использование аналитики

### Вкладка "Аналитика" в CRM
1. Зайдите в раздел **CRM**
2. Переключитесь на вкладку **"Аналитика"**
3. Изучите метрики:
   - Воронку продаж
   - Срезы по этапам
   - Время на этапах
   - Общую статистику

### Интерпретация данных

**Воронка продаж:**
- Широкий верх → много новых лидов
- Узкое горлышко → проблемный этап, нужно улучшить
- Высокая конверсия (>70%) → зеленый цвет
- Средняя (50-70%) → желтый
- Низкая (<50%) → красный

**Срезы:**
- Если много срезов на одном этапе → изучить причину
- Примеры проблем:
  - Срезы на "Оплата за растаможку" → возможно высокая цена
  - Срезы на "Дал реквизиты" → долгая подготовка документов

**Время на этапах:**
- Долгое время на этапе → нужно оптимизировать процесс
- Короткое время → эффективная работа

## 🗄️ База данных

### Новая колонка в таблице `leads`
```sql
stage_history TEXT -- JSON массив истории
```

### Формат хранения
История хранится как JSON string:
```json
[
  {
    "id": "hist-123",
    "timestamp": "2026-02-04T10:00:00Z",
    "previousStage": null,
    "newStage": "contract_done",
    "changedBy": "manager-1",
    "notes": "Создан лид"
  },
  {
    "id": "hist-124",
    "timestamp": "2026-02-05T14:30:00Z",
    "previousStage": "contract_done",
    "newStage": "gave_requisites",
    "changedBy": "teamlead-1",
    "notes": "Выдал реквизиты"
  }
]
```

## 🚀 Миграция существующей БД

Если у вас уже есть лиды в базе данных, выполните миграцию:

```sql
-- Запустите файл .same/migration-add-stage-history.sql в Supabase SQL Editor
```

Миграция:
1. Добавит колонку `stage_history` в таблицу `leads`
2. Инициализирует историю для всех существующих лидов
3. Создаст первую запись с текущим статусом каждого лида

## 💡 Рекомендации

### Для документалистов
- Создавайте лидов с правильным статусом "Договор сделан"
- Указывайте корректного менеджера
- Заполняйте заметки при создании

### Для тимлидов
- Своевременно обновляйте статусы лидов
- При переводе в "Потеряно" указывайте причину в заметках
- Анализируйте срезы еженедельно

### Для владельца
- Следите за общей конверсией
- Сравнивайте эффективность команд
- Принимайте решения на основе данных

## 📊 Примеры использования

### Выявление проблемных этапов
```
Аналитика показала:
- 60% лидов теряются на этапе "Оплата за растаможку"
- Среднее время на этом этапе: 5 дней

Решение:
- Проверить адекватность цен
- Улучшить коммуникацию с клиентами
- Предложить альтернативные варианты оплаты
```

### Оптимизация процессов
```
Данные:
- Среднее время от "Договор" до "Завершено": 15 дней
- Команда Вохи: 12 дней
- Команда Зета: 18 дней

Действие:
- Изучить практики команды Вохи
- Применить best practices в команде Зета
```

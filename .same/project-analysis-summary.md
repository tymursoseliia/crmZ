### 2️⃣ ФИНАНСОВЫЙ МОДУЛЬ

#### Организационная структура:

**Команда Вохи:**
- Тимлид Воха
- Менеджеры команды

**Команда Зета:**
- Тимлид Зет
- Менеджеры команды

**Офис:**
- Жигуль
- Клыча
- Клоузеры (Валера и др.)

**IT отдел:**
- Разработчики с фиксированной ставкой

**Специальные роли:**
- Фотошоп (бонус 2500 USDT)
- Звук (исключен из расчетов по состоянию на февраль 2026)

#### Финансовые операции:

**Доходы:**
- **Растаможка** - основной тип операций
- **Добив** - дополнительные услуги

**Расходы (4 типа):**
1. **Персональные** - привязаны к конкретной команде (100% на команду)
2. **Технические** - делятся: 50% офис, 25% команда Вохи, 25% команда Зета
3. **Постоянные** - делятся 50/50 между командами
4. **Общие** - делятся 50/50 между командами

#### Система расчета зарплат:

**Менеджеры и Тимлиды:**
- % от растаможек (индивидуальная ставка, обычно 15%)
- % от добивов (обычно 10%)
- **Минимальная гарантия: 350 USDT**
- Формула: `MAX(350, процент от операций)`

**Клоузеры:**
- % от добивов (индивидуальная ставка)
- Некоторые имеют минималку (например, Валера - 250 USDT)

**Жигуль и Клыча:**
- % от растаможек и добивов
- Из зарплаты вычитается 50% их персональных расходов

**IT отдел:**
- Фиксированная месячная ставка
- Стоимость делится 50/50 между командами Вохи и Зета

**Фотошоп:**
- Фиксированный бонус 2500 USDT
- Вычитается из общей прибыли компании

#### Формула чистой прибыли:

```
Эффективная выручка = Выручка + Компенсация Бубона (50% тех. расходов)
Все расходы = Расходы из таблицы + Зарплаты всех сотрудников
Чистая прибыль = Эффективная выручка - Все расходы - Бонус Фотошопа (2500)
```

#### Функции модуля:

- **Dashboard** - общая статистика, KPI, графики
- **Операции** - учет растаможек и добивов
- **Расходы** - детальный учет по категориям
- **Сотрудники** - база данных с настройками зарплат
- **Расчет ЗП** - автоматический расчет по периодам
- **Отчеты** - аналитика, прогнозы, графики (Recharts)
- **KPI метрики** - целевые показатели
- **Экспорт в Excel** - выгрузка данных

### 3️⃣ CRM СИСТЕМА - ВОРОНКА ПРОДАЖ

#### Концепция:
Точное отслеживание **где** и **почему** теряются клиенты на пути от договора до завершения.

#### Этапы воронки (последовательные):

1. **Договор сделан** (начальный статус)
2. **Дал реквизиты**
3. **Оплата за растаможку**
4. **Оплата за авто**
5. **Оплата за утиль**
6. **Оплата за госпошлину**
7. **Оплата за залоговый платеж**
8. **Оплата прочее**
9. **Завершено** ✅ (финальный успех)
10. **Потеряно** ❌ (финальный провал - может произойти на любом этапе)

#### История статусов:

Каждое изменение статуса записывается с:
- Дата и время
- Предыдущий статус → Новый статус
- Кто изменил (менеджер/тимлид)
- Комментарий
- Сумма оплаты (для этапов оплаты)

#### Ключевая фича - Анализ срезов:

**Проблема:** Формальный статус лида может не совпадать с реальным этапом, на котором произошла потеря.

**Пример:**
```
Лид формально находится на: "Оплата за растаможку"
Но реально потерялся на этапе: "Дал реквизиты"
Причина: ждал реквизиты 5 дней, нашел конкурента
```

**Решение:**
При переводе в статус "Потеряно" обязательно указывается:
1. **lostAtStage** - на каком этапе РЕАЛЬНО произошел срез
2. **lostReason** - причина из 7 категорий
3. **lostReasonText** - подробное описание

#### 7 категорий причин потери:

1. 💰 **Высокая цена** - клиент нашел дешевле
2. ⏰ **Долгое ожидание** - слишком долго ждал
3. 🏪 **Нашел конкурента** - ушел к другой компании
4. 🤔 **Передумал** - изменил решение о покупке
5. 💸 **Нет денег** - не может оплатить
6. 📞 **Не отвечает** - пропал, не берет трубку
7. ❓ **Другое** - прочие причины

#### Аналитика:

**Воронка продаж:**
- Количество лидов на каждом этапе
- Процент от общего числа
- Конверсия между этапами (цветовая индикация)
- Потенциальная выручка на каждом этапе

**Анализ срезов:**
- Статистика: на каких этапах РЕАЛЬНО теряются лиды
- Визуализация проблемных зон
- Сравнение формальных и фактических потерь

**Анализ причин:**
- Топ причин потери клиентов
- Распределение по 7 категориям
- Процентное соотношение

**Среднее время на этапах:**
- Сколько дней лид проводит на каждом этапе
- Помогает выявить узкие места
- Метрика скорости работы команды

**Статистика команд:**
- Сравнение эффективности команд Вохи и Зета
- Конверсия, средний чек, скорость обработки

#### Валидация и контроль:

✅ **Разрешено:**
- Переход на следующий статус
- Переход в "Потеряно" с любого этапа
- Указание комментариев

❌ **Запрещено:**
- Перепрыгивать через этапы
- Возвращаться назад по воронке
- Переводить в "Потеряно" без указания причины
- Оплату без указания суммы
- Изменение финальных статусов (Завершено/Потеряно)

#### Real-time синхронизация:

- Использование Supabase subscriptions
- Автоматическое обновление при изменениях
- Работа в оффлайн режиме с localStorage

### 5. Двойное хранение данных:

**Приоритет:** Supabase → localStorage

```typescript
// Попытка загрузить из Supabase
const { data } = await supabase.from('operations').select();
if (data) {
  setOperations(data);
  localStorage.setItem('operations', JSON.stringify(data));
} else {
  // Fallback на localStorage
  const cached = localStorage.getItem('operations');
  if (cached) setOperations(JSON.parse(cached));
}
```

### Роли пользователей:

**Документалист:**
- Создает лидов
- Заполняет базовую информацию
- Устанавливает начальный статус

**Тимлид (Воха, Зет):**
- Управляет статусами лидов
- Выдает реквизиты
- Отслеживает оплаты
- Указывает причины потерь

**Владелец:**
- Анализирует статистику
- Сравнивает команды
- Принимает решения

### Процесс работы с лидом:

1. **Создание** - документалист
2. **Выдача реквизитов** - тимлид
3. **Отслеживание оплат** - тимлид (8 этапов)
4. **Завершение** - автоматически при оплате всего
5. **Потеря** - тимлид с указанием причины и реального этапа среза
